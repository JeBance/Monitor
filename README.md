## Видение проекта "Monitor"

### Концепция
**Monitor** — это гибкая, настраиваемая и безопасная платформа для создания персональных дашбордов, работающая полностью на стороне клиента и размещаемая на GitHub Pages.  
Основная идея: дашборд состоит из **виджетов**, которые наполняются данными через **плагины**. Плагины не содержат исполняемого кода — только JSON-описания, что обеспечивает высокий уровень безопасности.

### Цели
- Создать среду, где любой пользователь может добавлять новые источники данных и визуализации без риска XSS или внедрения вредоносного кода.
- Обеспечить простой механизм распространения плагинов через GitHub или любой статический хостинг.
- Сохранить возможность кастомизации внешнего вида и поведения виджетов в безопасных рамках (через схему и ограниченные шаблоны).

---

## Архитектура системы

### 1. Ядро (Core)
- Работает в браузере, загружается с GitHub Pages.
- Отвечает за:
  - Управление плагинами (загрузка манифестов, валидация по JSON Schema).
  - Рендеринг доски с виджетами (сетка, drag & drop, изменение размеров).
  - Жизненный цикл виджетов (создание, обновление данных, удаление).
  - Сохранение/восстановление состояния доски (LocalStorage, файл).
  - Безопасную обработку данных (санитизация, CSP).
- Ядро предоставляет **базовые типы виджетов** (текст, таблица, график, индикатор), которые можно использовать в плагинах. Типы реализованы как предопределенные React/Vue компоненты, принимающие данные и настройки.

### 2. Плагины
- Набор статических файлов (JSON + опционально иконки), доступных по URL.
- Содержат **манифест** (`manifest.json`) и конфигурации отдельных виджетов.
- Плагин может быть размещён в любом публичном репозитории (GitHub, GitLab) или на любом сервере с поддержкой CORS.
- Добавление плагина в дашборд происходит по ссылке на его манифест.

### 3. Виджеты
- Экземпляры, созданные на основе конфигурации из плагина.
- Имеют размер, позицию, собственное состояние (если нужно).
- Периодически запрашивают данные из внешних источников (REST API) согласно настройкам.

## Структура плагина (JSON-схемы)

### Манифест плагина (`manifest.json`)
```json
{
  "id": "unique-plugin-id",
  "name": "Название плагина",
  "version": "1.0.0",
  "description": "Краткое описание",
  "author": "Автор",
  "widgets": [
    {
      "id": "weather",
      "name": "Погода",
      "description": "Текущая погода в городе",
      "icon": "icons/weather.svg",
      "configUrl": "widgets/weather/config.json",
      "sizes": ["small", "medium"]   // допустимые размеры
    }
  ]
}
```

### Конфигурация виджета (`config.json`)
```json
{
  "type": "text",                // один из базовых типов
  "title": "Погода в {{city}}",  // можно использовать переменные из настроек
  "updateInterval": 1800,        // секунд, 0 — без автообновления
  "settingsSchema": {             // схема для пользовательских настроек (JSON Schema)
    "type": "object",
    "properties": {
      "city": { "type": "string", "default": "Moscow" },
      "units": { "type": "string", "enum": ["metric", "imperial"] }
    }
  },
  "dataSource": {
    "url": "https://api.openweathermap.org/data/2.5/weather?q={{city}}&units={{units}}&appid={{apiKey}}",
    "method": "GET",
    "headers": {},                // только статические, без авторизации (или через ключи)
    "transform": {
      "type": "jsonpath",         // или "jq", но безопаснее встроенные методы
      "expression": "$.main.temp"
    }
  },
  "template": {
    "html": "<div>{{temperature}}°C, {{description}}</div>",
    "sanitize": true              // включить санитизацию
  },
  "styles": {
    "backgroundColor": "#f0f0f0",
    "borderRadius": "8px"
  }
}
```

**Пояснения:**
- `dataSource` описывает, откуда и как получать данные. Поддерживаются подстановки параметров (из пользовательских настроек) через `{{...}}`.
- `transform` позволяет извлечь нужные поля из JSON-ответа. Можно реализовать несколько встроенных трансформеров: `jsonpath`, `pick` (выборка полей), `format` (конкатенация).
- `template` определяет HTML-разметку с плейсхолдерами. Шаблонизация должна быть безопасной: либо очень простой `{{field}}` с заменой, либо использовать Handlebars, но с ограниченным набором helpers и обязательной санитизацией результата (DOMPurify). Альтернатива — задавать не HTML, а ссылку на предопределённый тип (`text`, `table`), который ядро отрендерит само, без шаблона.
- Все поля валидируются по схеме на этапе загрузки.

## Безопасность

### 1. Отсутствие исполняемого кода в плагинах
- Плагины содержат только JSON. Любой JavaScript в них будет проигнорирован (если только не загружен через поле `script`, но такое поле не должно быть в схеме).
- Даже если злоумышленник подменит конфигурацию, максимум, что он сможет — изменить URL источника данных или HTML-шаблон. Но HTML будет санитизирован.

### 2. Санитизация вывода
- Все данные, вставляемые в DOM, проходят через `DOMPurify` (или аналоги). Теги `<script>`, обработчики событий и опасные атрибуты удаляются.
- Шаблоны, загруженные из конфига, также санитизируются перед рендерингом.

### 3. Валидация по JSON Schema
- Манифесты и конфиги проверяются по строгим схемам. Это исключает добавление неожиданных полей, которые могли бы интерпретироваться как код или изменять поведение ядра.
- Схемы описывают только разрешённые типы данных, структуру и ограничения (например, протокол URL только HTTPS).

### 4. Контроль источников данных
- URL в `dataSource` проверяются на соответствие протоколу (только `https:`) и опционально — на принадлежность к белому списку доменов (можно задать глобально в настройках дашборда).
- Можно добавить предупреждение при попытке использовать не HTTPS источник или локальный адрес.
- Для работы с API, требующими ключи, рекомендуется передавать их через переменные, которые пользователь вводит в настройках виджета (без сохранения в плагине).

### 5. Content Security Policy (CSP)
- GitHub Pages позволяет задавать HTTP-заголовки через `_headers` файл. Настроим строгую политику: запрет на `unsafe-inline`, `unsafe-eval`, разрешим только загрузку данных по HTTPS с ограниченных доменов (можно ослабить для плагинов, но лучше разрешить все https, т.к. плагины могут использовать любые API).

### 6. Изоляция iframe (опционально)
- Если потребуются более сложные виджеты (например, интерактивные карты), можно создать тип "iframe", который будет загружать сторонний контент в изолированном `sandbox` iframe с параметрами `allow-same-origin`, `allow-scripts` (только если доверяете источнику). Но по умолчанию этот тип отключён или требует явного подтверждения от пользователя.

## Интерфейс пользователя

- **Дашборд** — сетка (CSS Grid или библиотека типа React-Grid-Layout). Виджеты можно перетаскивать, изменять размер, удалять.
- **Боковая панель** со списком установленных плагинов и доступных виджетов. Кнопка "Добавить плагин" (ввод URL манифеста).
- **Режим настройки**: при клике на виджет открывается панель с его настройками, определёнными в `settingsSchema` (генерируется автоматически из схемы).
- **Сохранение**: кнопки "Сохранить доску" (экспорт в JSON) и "Загрузить" (импорт). Автосохранение в LocalStorage.

## Технический стек

- **Фреймворк**: React (или Preact для лёгкости) — удобно для компонентного подхода.
- **Управление состоянием**: Redux Toolkit / Zustand (для хранения плагинов, виджетов, настроек).
- **Сетка**: react-grid-layout (или подобная).
- **Валидация JSON**: Ajv (или просто встроенный `JSON.parse` + проверка по схеме).
- **Санитизация**: DOMPurify.
- **Шаблонизация**: Минимальный самодельный интерполятор (без eval) или Handlebars с отключёнными опасными возможностями.
- **HTTP-запросы**: Fetch API.

## Процесс работы

1. **Инициализация**: загружается ядро, читает сохранённое состояние (если есть).
2. **Управление плагинами**:
   - При добавлении плагина ядро загружает `manifest.json` по указанному URL.
   - Проверяет его по схеме, извлекает список виджетов.
   - Сохраняет информацию о плагине в хранилище.
3. **Создание виджета**:
   - Пользователь перетаскивает виджет из панели на доску.
   - Ядро загружает конфиг виджета по `configUrl` (относительно плагина), валидирует.
   - Если в конфиге есть `settingsSchema`, открывается диалог для ввода параметров.
   - Виджет добавляется в сетку с параметрами по умолчанию.
4. **Обновление данных**:
   - Для каждого виджета запускается таймер (согласно `updateInterval`).
   - При срабатывании формируется URL с подстановкой параметров (и, возможно, API-ключа из настроек пользователя).
   - Выполняется `fetch` с учётом CORS.
   - Ответ обрабатывается трансформером.
   - Полученные данные прогоняются через санитайзер и вставляются в шаблон.
   - Виджет перерисовывается.
5. **Редактирование виджета**: двойной клик/кнопка "настройки" открывает форму, созданную динамически на основе `settingsSchema`. После сохранения — обновление данных.

## Распространение и экосистема

- Плагины могут храниться в GitHub-репозиториях. Пользователь указывает URL к raw-версии `manifest.json` (например, `https://raw.githubusercontent.com/user/plugin/main/manifest.json`).
- Можно создать каталог плагинов (отдельный репозиторий), где собраны ссылки на проверенные плагины. Ядро может загружать этот каталог и предлагать плагины для установки.
- Версионирование плагинов: при изменении файлов на сервере пользователь может обновить плагин вручную (кнопка "проверить обновления").

## Ограничения и вызовы

- **CORS**: многие публичные API не разрешают запросы с произвольных источников. Пользователям придётся либо использовать API с поддержкой CORS, либо настраивать прокси. GitHub Pages не предоставляет серверного прокси, поэтому это ограничение остаётся. Можно рекомендовать использовать сервисы вроде `cors-anywhere` (но это небезопасно) или поощрять API с открытым CORS.
- **Безопасность API-ключей**: ключи, введённые пользователем в настройках виджета, хранятся только в LocalStorage (или не хранятся, а запрашиваются каждый сеанс). Это не очень удобно, но безопасно. Можно хранить зашифрованными с мастер-паролем, но это усложняет.
- **Сложность трансформации данных**: простые JSONPath выражения могут не покрыть все нужды. Можно добавить возможность писать простые функции на безопасном языке (например, Jsonata), но это потенциально может привести к атакам, если позволить выполнение произвольных выражений. Jsonata безопасна, т.к. не имеет доступа к окружению, но её парсинг и выполнение — дополнительный риск. Лучше начать с набора встроенных операций (pick, map, filter).
- **Размер плагинов**: конфиги могут быть большими, но это JSON, обычно не проблема.

## Пример использования

1. Пользователь хочет видеть курс биткоина.
2. Находит плагин "Crypto rates" с виджетом "Bitcoin price".
3. Добавляет плагин по URL.
4. Перетаскивает виджет на доску.
5. В настройках вводит свой API-ключ (если нужно) и выбирает валюту.
6. Виджет начинает показывать актуальную цену, обновляясь раз в минуту.

## Заключение

Проект **Monitor** может стать удобной и безопасной альтернативой сложным самописным дашбордам. Основной упор на безопасность через декларативные JSON-плагины и строгую валидацию позволяет привлекать сторонних разработчиков без риска для пользователей. Благодаря размещению на GitHub Pages проект легко форкнуть, доработать или разместить собственный экземпляр.

Дальнейшие шаги:
- Разработать прототип ядра с базовыми типами виджетов.
- Создать документацию по написанию плагинов.
- Разработать примеры плагинов для популярных API (погода, курсы, статистика GitHub).
- Рассмотреть возможность добавления поддержки WebSocket для real-time данных (с дополнительными мерами безопасности).
